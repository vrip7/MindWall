## Reporting a Vulnerability

**Do NOT open a public GitHub issue for security vulnerabilities.**

Instead, please report security issues responsibly:

1. **Email:** Send a detailed report to **contact@vrip7.com**
2. **Subject line:** `[MindWall Security] Brief description`
3. **Include:**
   - Description of the vulnerability
   - Steps to reproduce
   - Potential impact assessment
   - Suggested fix (if any)

### Response Timeline

| Stage | Timeline |
|-------|----------|
| Acknowledgment | Within 48 hours |
| Initial assessment | Within 5 business days |
| Fix development | Within 30 days (critical), 90 days (non-critical) |
| Public disclosure | After fix is released and deployed |

## Security Architecture

### Deployment Model

MindWall is designed for **internal network deployment only**. All components run within a single Docker Compose stack on an organization's private infrastructure.

### Key Security Properties

| Property | Implementation |
|----------|---------------|
| **Data isolation** | Zero external API calls — all LLM inference runs locally via Ollama |
| **Network boundary** | Dual Docker networks: `mindwall-internal` (Ollama + API, no internet) and `mindwall-host` (API + proxy + dashboard) |
| **Authentication** | API key authentication via `X-MindWall-Key` header with constant-time `hmac.compare_digest()` validation |
| **Dashboard auth** | Username/password login via `POST /auth/login` returning API key |
| **Secret management** | Secrets auto-generated by setup scripts, stored in `.env` (not committed to git) |
| **Database** | SQLite — file-based, no network-exposed database server |
| **TLS** | Proxy accepts plaintext on localhost, opens TLS connections to upstream mail servers |
| **Request tracing** | UUID4 `X-Request-ID` header on every API request for end-to-end tracing |
| **Data privacy** | Email bodies processed in-memory and discarded — only analysis results persisted |

### Port Exposure

| Port | Service | Protocol |
|------|---------|----------|
| 5297 | API Engine | HTTP + WebSocket |
| 4297 | Dashboard | HTTP |
| 1143 | IMAP Proxy | IMAP |
| 1025 | SMTP Proxy | SMTP |
| 11434 | Ollama | Internal only — **not exposed to host** |

### Hardening Recommendations

1. **Network isolation:** Deploy on a dedicated VLAN or subnet
2. **Firewall rules:** Only allow internal clients to reach ports 1143, 1025, 4297, 5297
3. **Never expose to internet:** MindWall ports should not be publicly accessible
4. **Rotate API keys:** Regenerate `API_SECRET_KEY` periodically
5. **Change default credentials:** Update `DASHBOARD_USERNAME` and `DASHBOARD_PASSWORD` from defaults
6. **Update regularly:** Keep Docker images, NVIDIA drivers, and Ollama updated
7. **Monitor logs:** Structured JSON logging via structlog — forward to your SIEM
8. **Restrict Docker:** Non-root containers are already configured
9. **Backup:** Regularly back up `data/db/mindwall.db`
10. **Disk encryption:** Consider full-disk encryption for sensitive deployments (SQLite is not encrypted at rest)

### What MindWall Stores

| Data | Location | Purpose |
|------|----------|--------|
| Email metadata | `analyses` table | Analysis records (subject, sender, recipient, scores) |
| Manipulation scores | `analyses` table | Risk assessment and dimension breakdowns |
| LLM explanations | `analyses` table | Human-readable analysis for dashboard review |
| Sender baselines | `sender_baselines` table | Behavioral deviation detection (EMA patterns) |
| Email credentials | `email_accounts` table | Upstream IMAP/SMTP authentication (plaintext, required for proxy) |

### What MindWall Does NOT Do

- Does not send any data to external servers
- Does not phone home or collect telemetry
- Does not persist full email bodies — processed in-memory, discarded after analysis
- Does not log email body content or passwords
- Does not process email attachments — MIME parser skips binary content

### Limitations

- **No HTTPS on proxy** — Plaintext on localhost by design; wrap in TLS-terminating reverse proxy for remote deployments
- **No RBAC** — Single shared API key; no per-user roles or permissions
- **No encryption at rest** — SQLite database is unencrypted; use full-disk encryption on the host
- **Plaintext credential storage** — Email passwords stored unencrypted because the proxy requires plaintext for upstream authentication

## Scope

The following are **in scope** for security reports:

- Authentication bypass
- SQL injection in the API layer
- WebSocket hijacking
- IMAP/SMTP proxy vulnerabilities
- Container escape risks
- Sensitive data exposure in logs
- Dependency vulnerabilities

The following are **out of scope:**

- Attacks requiring physical access to the server
- Social engineering of administrators
- Denial of service on localhost services
- Issues in upstream dependencies with existing CVEs (report to upstream)
